# Vibe Code Report

## Findings

| ID | File | Line | Severity | Finding |
| --- | --- | ---: | --- | --- |
| VC-001 | `components/GridcraftStudio.tsx` | 56 | HIGH | The main client component keeps more than a dozen `useState` slices for one workflow. Input, command, formats, output, error, loading, sharing, history, and sidebar state are all managed independently, which makes inconsistent transitions much easier to introduce. |
| VC-002 | `components/GridcraftStudio.tsx` | 125 | HIGH | `clearRunState()` fans one conceptual state reset across six separate setters. That is a classic state-fragmentation smell and makes it easy for future edits to reset only part of the run state. |
| VC-003 | `components/GridcraftStudio.tsx` | 210 | MEDIUM | Input, command, and format change handlers all duplicate the same “mark custom preset + clear output” behavior. The duplication is small but already broad enough to encourage drift. |
| VC-004 | `components/GridcraftStudio.tsx` | 235 | HIGH | `handleCommandKeyDown()` mixes submit logic, history navigation, and command restoration in one long branchy handler. It is doing too many things for a hot interaction path and is easy to regress. |
| VC-005 | `components/GridcraftStudio.tsx` | 282 | HIGH | `handleRun()` owns validation, URL syncing, history recording, loading transitions, network execution, metrics, preview generation, and error handling in one function. It is the critical path and already too large and branch-heavy to maintain safely. |
| VC-006 | `lib/runTransform.ts` | 18 | HIGH | The fetch layer has no `AbortController`, no timeout, and no request sequencing. If a request hangs or a newer request supersedes an older one, there is no cancellation path and no deterministic winner. |
| VC-007 | `components/GridcraftStudio.tsx` | 314 | HIGH | The execution path accepts the first resolved response and immediately commits it to UI state. There is no request id or stale-response guard, so overlapping requests can overwrite newer state if concurrency is ever introduced by double submission, future auto-run behavior, or route transitions. |
| VC-008 | `components/OutputPanel.tsx` | 75 | MEDIUM | Large results are still rendered as one monolithic `<pre>` string. Preview truncation helps, but the render path is not windowed and can still become expensive for long outputs. |
| VC-009 | `app/page.tsx` | 7 | MEDIUM | The suspense fallback is `null`, so users get a blank loading state during route hydration instead of a visible placeholder. This is a polished-looking omission that usually comes from getting the build to pass as quickly as possible. |
| VC-010 | `app/api/run/route.ts` | 133 | HIGH | The API response shape is ad hoc and not code-based. User mistakes, server failures, and policy violations all collapse into `{ output, error }`, which makes precise client handling and observability harder than it should be. |
| VC-011 | `app/api/run/route.ts` | 137 | MEDIUM | Request validation is handwritten and partial. It covers the current form but does not give the route a single runtime schema for request shape, limits, and error codes. |
| VC-012 | `app/api/run/route.ts` | 194 | MEDIUM | The route only checks that the engine file exists, not that it is actually executable. A permissions problem would fail later and less clearly than necessary. |
| VC-013 | `lib/commandPolicy.ts` | 48 | MEDIUM | The blocked DSL helper check runs against the raw command string with a broad regex. That is simple, but it can reject benign string literals and is not truly syntax-aware. |
| VC-014 | `lib/shareState.ts` | 56 | MEDIUM | The app serializes the entire workspace into the URL on every run. This makes sharing convenient for small inputs but creates brittle long-URL behavior for larger datasets and repeats heavy encoding work on the client. |
| VC-015 | `components/GridcraftStudio.tsx` | 351 | LOW | Clipboard failure only changes the button label. There is no dedicated status region, so assistive feedback is weaker than it should be for an important action. |
| VC-016 | `components/OutputPanel.tsx` | 66 | LOW | The error surface uses color plus text, but the general output status region and action feedback are mixed into one area. This works, but it is not a clean accessibility boundary. |
| VC-017 | `components/GridcraftStudio.tsx` | 116 | LOW | `selectedPreset` is derived from `selectedPresetId` on every render and immediately used only for description text. This is minor, but it is an example of derived UI lookup being left inline in a component that already does far too much. |
| VC-018 | `components/InputPanel.tsx` | 1 | LOW | The presentational panel is pure but not memoized, so it re-renders whenever the parent changes unrelated state such as copy feedback or sidebar visibility. |
| VC-019 | `components/CommandBar.tsx` | 1 | LOW | The command bar is also a pure presentational component receiving stable scalar props, but it is not memoized and will re-render on unrelated parent state changes. |
| VC-020 | `components/OperationsReference.tsx` | 1 | LOW | The operations sidebar is pure, renders a fairly large static list, and is not memoized or dynamically loaded. It is an easy place to reclaim unnecessary work from the first-load path. |
| VC-021 | `components/OutputPanel.tsx` | 3 | LOW | `RunSummary` is redefined locally instead of shared from a single source. This is a small copy-paste smell that grows maintenance cost over time. |
| VC-022 | `scripts/prepare-engine.mjs` | 20 | MEDIUM | The postinstall script renames any lone file in `bin/` to the engine name. That assumption is convenient but brittle and can silently accept the wrong artifact. |
| VC-023 | `tsconfig.json` | 1 | MEDIUM | Tooling mutates the TypeScript config to inject `.next/types/**/*.ts`, which means linting and typechecking behavior depends on prior build state. That is exactly the kind of environment-coupled shortcut that becomes painful in CI and clean checkouts. |
| VC-024 | `next.config.mjs` | 1 | MEDIUM | The app has no security headers, no content security policy, and no asset cache policy configuration. For a local prototype that is common; for a production app it is unfinished. |
| VC-025 | `app/layout.tsx` | 1 | MEDIUM | There is no error boundary anywhere in the component tree. A render-time failure in the main client surface would currently take down the whole page. |
